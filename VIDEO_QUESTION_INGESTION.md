# 動画由来の何切る問題 取り込み規約

短い動画（目安1〜3分）から問題を追加する時の、再現可能な作業手順と保存規約です。字幕の誤変換に左右されず、将来の牌画像置換と m/p/s のランダム入替にも耐えるデータを残します。

## 必ず守る保存原則

### 牌は正規コードで保存する

問題データ、解説文、注釈、正解牌には、次の ASCII 牌コードだけを使います。

| 種類 | 保存コード |
| --- | --- |
| 萬子 | `1m`〜`9m` |
| 筒子 | `1p`〜`9p` |
| 索子 | `1s`〜`9s` |
| 字牌 | `1z` 東、`2z` 南、`3z` 西、`4z` 北、`5z` 白、`6z` 發、`7z` 中 |
| 赤5 | `0m`、`0p`、`0s` |

- 解説の結論は `打6m。`、`打4zリーチ。` のように書く。
- 複数牌は必ず一牌ずつ書く。`3s・6s・9s待ち`、`3p・4p・5pのチー` のようにし、`3・6・9索`、`345p`、`六萬` は使わない。
- `萬`・`万`・`筒`・`索`、Unicode 牌、牌画像の HTML、Markdown 画像は `explanation` と `note` に保存しない。
- 点数・符・局名など、牌ではない数値（`1300・2600`、`40符`、`東1局` など）は通常の日本語表記のままでよい。
- 既存の説明に漢字牌表記が残っている場合は、その問題を触るタイミングで正規コードへ直す。今回追加した第166・167問はすでにこの規約に合わせる。

この正規コードは**データの原本**です。将来の表示側は、解説文中の個別コードを牌画像へ変換するだけにし、JSONの `hand`、`correctDiscards`、成績、保存済み回答を表示用の牌名へ書き換えません。

### 将来の m/p/s ランダム入替

- 1問を描画する時にだけ、m/p/s の置換表（例: `m→p, p→s, s→m`）を一つ決める。
- 手牌、ドラ、副露、選択肢、解説内の牌画像には、同じ置換表を使う。`0m` は対応先の赤5へ置換する。
- `z` は入れ替えない。
- 正誤判定、JSON、localStorage、管理画面の編集値は常に正規コードのままにする。表示用に入れ替えたコードを保存しない。
- 牌コードを一牌ずつ明記しておくことで、後のレンダラーは文字列を安全にトークン化して画像に置き換えられる。

## 動画からの取り込み手順

1. 動画URL、動画タイトル、公開日、問題番号を記録する。問題フレームを `public/questions/question-NNN.png` として保存する。
2. 問題表示、局面表示、ドラ表示、副露表示、結論表示の各フレームを目視確認する。自動字幕・文字起こしは確認箇所を探す補助に使うだけで、確定根拠にしない。
3. `public/questions.json` に問題を追加する。画像が複数なら `images` にすべて保存し、`image` は先頭画像にする。
4. 解説は文字起こしの転載ではなく、結論を先頭に置いた短い要約にする。正解、待ち、選択理由、代替案の不利を、動画で確認できる範囲だけで説明する。
5. `scripts/structure-questions.py` の手動上書きとテストを更新し、構造化結果と表示を確認する。

## 構造化データの規約

最低限、次を確認します。

```json
{
  "id": 168,
  "image": "/questions/question-168.png",
  "images": ["/questions/question-168.png"],
  "explanation": "打6m。\n\n6mを切ると…",
  "sourceUrl": "https://youtu.be/example",
  "sourceLabel": "YouTube動画を開く",
  "createdAt": "2026-07-17T00:00:00Z",
  "course": "intermediate",
  "hand": ["1m", "2m"],
  "draw": null,
  "meldCount": 0,
  "round": "east1",
  "seat": "west",
  "turn": 6,
  "honba": 0,
  "points": 25000,
  "dora": "6s",
  "melds": [],
  "correctDiscards": ["6m"],
  "status": "unreviewed"
}
```

- `hand` には副露牌を入れない。`meldCount` が `n` の時、ツモを分離しない現行仕様では手牌は `14 - 3n` 枚である。
- 問題の編は `course` で分ける。初級編は `"beginner"`（省略時も初級編）、中級編は `"intermediate"` を保存する。今後追加する中級編の動画問題には必ず `"course": "intermediate"` を入れる。
- ツモ牌が別表示でも、現行アプリでは理牌して `hand` に入れ、`draw` は常に `null` にする。鳴いた直後でツモ牌がない場合も同じで、必要なら `note` に `6sをチーした直後（ツモ牌表記なし）` のように書く。
- `melds` には見えている副露をすべて入れる。一つだけ確認して確定しない。第167問のように複数チーがある場合も全面子を記録する。
- チーは上家からのみ行う。UIでは鳴いた牌を左端で横向きにするため、新規データは `{"type":"chi","open":true,"calledIndex":0,"tiles":["6s","4s","0s"]}` のように、鳴いた牌を最初に置く。
- ポン・カンは動画の横向き牌を確認し、実際に鳴いた牌の位置を `calledIndex` に記録する。
- リーチ判断が問題なら `riichiChoice: true` と `correctRiichi` を設定する。暗槓を選ばせる問題は、既存のカン選択肢の判定規約に従う。

## 字幕・文字起こしの扱い

自動字幕は麻雀用語と牌名を頻繁に誤認します。次のような文字列をそのまま解説へ貼り付けません。

- `六萬`、`ローマン`、`ろくまん` のような表記は、牌姿と文脈を確認してから `6m` にする。
- `中`、`白`、`發`、風牌は一般語としても現れる。動画の牌表示と照合し、牌を指す場合だけ `7z`、`5z`、`6z`、`1z`〜`4z` にする。
- `聴牌`、`フリテン`、`一盃口`、`平和`、`タンヤオ`、`リーチ`などは麻雀用語として正しい漢字・カタカナに直す。ただし動画で確認できない打点、残り枚数、役は補完しない。
- 字幕と牌姿、結論フレームのいずれかが食い違ったら、字幕ではなく動画フレームを優先する。副露・赤5・鳴いた牌の向きが曖昧なら、より上位の確認モデルへ回すか、人の確認を求める。

## 解説の書き方

- 先頭を結論にする。例: `打6m。`、リーチ判断があるなら `打4zリーチ。`
- 続けて「なぜその選択が最善か」と「代表的な代替案より優れる理由」を簡潔に書く。
- 牌は必ず個別の正規コードで書く。例: `3s・6s・9s待ち`、`4mを切る`、`7zが雀頭に残る`。
- 動画URLは `sourceUrl` に保存する。解説本文にURLや文字起こし全文を混ぜない。
- 断定できない部分は作らない。誤認の恐れがある待ち枚数・符・点数は省略し、確認できた事実だけを記載する。

## 検証と公開

1. `scripts/structure-questions.py` の `HAND_OVERRIDES`、`SITUATION_OVERRIDES`、`MELD_OVERRIDES` を必要に応じて更新する。動画フレームはOCRだけで確定しない。
2. `tests/admin-api.test.mjs` に問題番号、手牌、副露、ドラ、局面、正解、URL、正規牌表記の検証を足す。
3. `npm run structure:questions`、`npm test`、`git diff --check` を実行する。
4. PC幅とスマホ幅で、手牌・副露の横向き牌・ドラ・解説を確認する。
5. アプリのデータや表示を変えた場合は `APP_VERSION` を一つ上げ、テストの期待値も更新する。ドキュメントだけの変更なら上げない。
6. GitHub Pagesへ公開する変更はpush後、キャッシュを避けた公開ページでも確認する。

## Luna / Terra への引き継ぎ目安

- 手順が定型化され、牌姿・字幕・結論が一致している動画は、LunaやTerraでもこの文書をチェックリストとして処理できる。
- 字幕の崩れ、赤5、複数副露、カン・リーチ判断、待ち枚数・打点計算、動画フレーム間の矛盾がある場合は、より慎重な確認を行うモデルへ切り替えるか、人に確認する。
- モデルに関係なく、この文書の「動画フレームで確定」「正規牌コードで保存」「テストで固定」の三点は省略しない。

## 同一配信シリーズの校正セット

同じレイアウトの動画を継続して追加する場合は、`calibration/zundamon-nanikiru-v1/README.md` をこの文書と一緒に読む。`manifest.json` の既知問題画像を、問題フレームの配置、赤5、ドラ、副露、リーチ判断の照合例として使う。

- 校正セットはモデル間で共通だが、以前の推論結果を根拠にしてはならない。各問題は動画フレームで再確認する。
- `npm run validate:video-question -- <id>` は、中級編の動画問題について、牌コード、手牌枚数、副露数、画像、URL、解説表記を機械検証する。
- 同じ動画URLが既存の初級編にある場合は、二重登録せず初級編を残す。中級編へ移動するかどうかは明示的な指示がある場合だけ行う。
