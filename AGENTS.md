# AGENTS.md

このファイルは、エンスクドリルと同系統のアプリを今後制作・改修するエージェント向けの制作指針です。

目的は、単に色や部品をコピーすることではなく、エンスクドリルで成立した「速度重視の操作性」「高密度だが読みやすい情報設計」「アーケード寄りの達成演出」「スマホ実戦運用に耐えるUI」を再現することです。

## 基本方針

- 日本語UIを前提にする。
- スマホ利用を最優先に設計する。
- PCでも破綻しないが、主戦場は片手スマホ操作とする。
- ランディングページではなく、起動直後に使えるアプリ画面を作る。
- 操作速度、誤タップ防止、視線移動の少なさを最優先にする。
- 画面上の説明文は最小限にする。
- 余計なチュートリアル文や装飾文を増やさない。
- アプリ内の言葉は短く、機能名は明確にする。
- 実装判断に迷ったら、見た目の派手さより操作感を優先する。

## UIテイスト

エンスクドリルの方向性は「競技卓ミニマル」から始まり、最終的には「和風アーケード寄り」に寄せた。

次回以降も、近いテイストにする場合は以下を基準にする。

- 背景は深緑、黒緑、卓面感のある暗色をベースにする。
- うっすらグリッドや卓面の質感を入れる。
- 文字は強めの明朝体を基本にする。
- 見出しは大きく、太く、影を付けて力を出す。
- 角丸は使うが、丸すぎるカード乱用は避ける。
- メイン操作ボタンは濃いオレンジ、銅、アンティークゴールド系。
- 裏モードや高難度モードは濃い紫系。
- 選択中タブや重要な操作はアンティークゴールド系。
- 無効・未選択・補助ボタンは黒緑の半透明。
- ランクSSや神は特別扱いし、虹色や専用画像で差別化する。

## 色の役割

目安:

- 背景: `#06110f`, `#0b2f28`, `#0f493c`
- 文字: クリーム系 `#f6f2df`
- 補助文字: グレーグリーン系 `#9aa9a4`
- ゴールド: `#d8a83c`, `#f2d27b`
- 表モード/通常プレイ: オレンジ、銅、ブラウンゴールド
- 裏モード/特殊モード: 暗い紫
- 正解: 赤丸
- 不正解: 青バツ

重要:

- 一色だけで画面を作らない。
- 緑背景、金、オレンジ、紫、赤、青を役割ごとに使う。
- ただし彩度を上げすぎず、背景は暗く締める。

## フォント

基本は明朝体。

推奨:

```css
font-family: "Yu Mincho", "YuMincho", "Hiragino Mincho ProN", "HGS明朝E", "MS PMincho", serif;
font-weight: 900;
letter-spacing: 0;
```

補足:

- 文字に力がない場合は、フォントサイズだけでなく `font-weight`, `text-shadow`, `-webkit-text-stroke` を調整する。
- ボタン内の文字も明朝体で統一する。
- ヒーロー風の大見出し以外では、過度に巨大な文字を使わない。

## レイアウト

- 基本幅はスマホ基準で `max-width: 28rem` 前後。
- ただしPC表示時も中央寄せで破綻させない。
- メニューの主要タブは横並びで視認性を優先する。
- 小さいボタンが他のボタンと重ならないよう、PC幅でも検証する。
- 設定ボタンは補助ボタン扱い。メインタブと干渉させない。
- メニュー画面や分析画面は縦スクロール可。
- プレイ中は上下左右に動かないよう固定する。
- プレイ開始時は必ずスクロール位置を先頭に戻す。

## 操作性

エンスクドリルの最重要価値は操作性。

- プレイ中は誤スクロール、横ズレ、ダブルタップ拡大を防ぐ。
- 数字キーはタップしやすく、隙間を極力減らす。
- 片手操作を想定し、操作パネル位置を設定可能にする。
- ボタンサイズは最低でも「通常」「小さめ」を選べるようにする。
- 数字配列は標準配列とテンキー配列を選べるようにする。
- スライドタッチはデフォルトONにする。
- スライドタッチは、押したまま別ボタンへ移動しても選択できる操作方式。
- 速度重視アプリでは、確定ボタンの有無や位置をモードごとに最適化する。

## プレイ画面

- プレイ中はスクロールをロックする。
- 画面固定はプレイ画面だけに限定する。
- メニュー、結果、分析、一覧では縦スクロールを許可する。
- プレイ画面中に横スクロールが出る実装は禁止。
- 問題画像や主要表示は一画面内に収める。
- 問題切り替え時に画像の遅延や残像を出さない。
- 画像を使う場合は事前ロード、二重スロット描画、または表示切替前decodeを使う。
- 次の問題に切り替わるタイムラグは極力ゼロに近づける。
- 回答後の正誤表示は短く、邪魔にならない位置に出す。

## 結果発表演出

エンスクドリルの完成度を支えている重要要素。

今後のアプリでも、成績画面は単なる数値表示にしない。

基本順序:

```text
スコア/ゲージ上昇
→ ランク表示
→ 新記録ならハンコ
→ コメント表示
```

重要:

- ランクコメントを先に表示しない。ランク先バレになる。
- スコア確定前にランクを出さない。
- ゲージとスコアは連動させる。
- 高スコア帯までの到達に一定の緊張感を持たせる。
- 新記録時はハンコ、SE、軽い遅延表示で達成感を作る。

移植用の詳細は以下を参照:

```text
RESULT_ANIMATION_HANDOFF.md
```

## 記録更新ハンコ

記録更新ハンコは、成績発表演出の中でも特に再現精度を重視する。

- 記録更新時のハンコは必ず `assets/record-stamp-transparent-v2.png` を使う。
- 背景付き画像、黒い矩形が見える画像、古い候補画像で代用しない。
- ハンコはスコア円の右下に、円の上から押されるように重ねる。
- ハンコは円ゲージの内側に閉じ込めず、円の前面レイヤーとして表示する。
- サイズ、角度、位置、表示タイミング、効果音は `RESULT_ANIMATION_HANDOFF.md` の `.record-stamp`, `@keyframes stamp-hit`, `tone("stamp")` を原則そのまま使う。
- ハンコ表示は、スコア上昇とランク表示が完了した後に行う。
- ハンコ表示後にランクコメントを表示する。
- ハンコ効果音は短い「ドン」「デデン」系にし、長い音や軽すぎるクリック音にしない。

## ランク表示

ランク表示は、エンスクドリルの達成感を作る重要要素として扱う。

- `神` ランクは必ず `assets/god-rank.png` を使う。
- `神` ランクは通常テキストで代用しない。
- `SS` ランクは画像ではなく、虹色グラデーション文字として表示する。
- `S` ランクは金色文字として表示する。
- `A` 系は赤系、`B` 系は緑系、`C` は青系、`D/E` はグレー系にする。
- メニュー、成績発表、ランキング、自己分析でランク表現が食い違わないようにする。
- 成績発表画面の円形ゲージ、神画像、SS虹色表示、記録更新ハンコの詳細は `RESULT_ANIMATION_HANDOFF.md` を参照する。

現在のエンスクドリルでは、SS用の画像は使っていない。SSは虹色文字、神だけが専用画像。

## 音

- 音量設定を用意する。
- 音量は0をミュートとして扱う。
- 初期音量は極端に小さくしない。
- iPhone/Androidではスリープ復帰や初回タップ後のAudioContext復旧を考慮する。
- 重要操作には短いSEを入れる。
- ただし連打する操作音は不快にならない長さにする。

## 設定画面

設定画面は小さくてもよいが、実用性を重視する。

最低限あるとよい項目:

- ニックネーム
- 音量
- ボタンサイズ
- ボタン位置
- 数字配列
- スライドタッチ
- バックアップ保存
- バックアップ復元

設定項目にはプレビューを置く。プレビューは実際のボタンサイズ、位置、配列を反映する。

## データ保存

- 個人成績、設定、復習、お気に入りは基本的に `localStorage` 保存でよい。
- ランキングなど共有データだけ外部DBへ送る。
- 端末移行用にファイル保存・ファイル復元方式を用意する。
- バックアップには端末IDも含める。
- ただしユーザーに「どのデータが共有されるか」は明確にする。

## ランキング

- ユーザー登録なしで投稿できる設計は、利用ハードルが低い。
- 投稿名は設定画面のニックネームを使う。
- 投稿後は送信済みにして連打投稿を防ぐ。
- 投稿直後に当日順位を返す。
- 同一名称は最高記録だけ表示する。
- 明らかなテスト名やNG名は弾けるようにする。
- Supabaseなどを使う場合、フロントに置いてよいのはpublishable/anon keyのみ。
- `service_role` やsecret keyは絶対に公開クライアントへ置かない。

## 自己分析・一覧

エンスクドリルでは、遊んだ後に成長を見られることが継続率に効いている。

今後のアプリでも、可能なら以下を用意する。

- 自己分析
- 日別ベスト推移
- 自己記録更新履歴
- モード別成績
- 問題一覧
- 問題ごとの正答率
- 苦手順ソート

グラフは単に全点を表示するのではなく、成長が見えるようにスケールを調整する。

## 画像・アセット

麻雀牌を使う場合は、必ずプロジェクト指定の牌画像を使う。

エンスクドリルでは以下を使用している。

```text
tiles/
```

別の麻雀アプリでも、原則として絵文字牌や外部牌画像で代用しない。

## 動画由来の問題追加

1〜3分程度の動画から何切る問題を追加・修正する時は、必ず `VIDEO_QUESTION_INGESTION.md` を読む。

- 自動字幕は手がかりにとどめ、牌姿・副露・局面・ドラ・正解は必ず動画フレームで確定する。
- `explanation`、`note`、正解牌などの保存データでは、牌を ASCII の正規牌コード（例: `3m`、`6p`、`9s`、`4z`、赤5は `0m`）で表記する。萬・筒・索の漢字、牌絵文字、HTMLの画像タグを保存しない。
- 一つずつの牌をそれぞれコード化する。たとえば待ちは `3s・6s・9s` と書き、`3・6・9s` や `369索` のように省略しない。将来、解説内の牌コードだけを牌画像へ置換し、m/p/sをランダムに入れ替えられるようにするためである。
- チーは上家からの牌を左端で横向きに表示する。新規・修正時のチー面子は、鳴いた牌を配列の先頭に置き、`calledIndex: 0` とする。
- `hand` は副露を除く理牌済みの手牌だけを保存し、ツモ牌も手牌へ混ぜて `draw: null` とする。副露の見落としを防ぐ。

## 実装方針

- 既存コードの流れを読んでから編集する。
- 変更範囲を小さく保つ。
- UI変更後はPC幅とスマホ幅を両方確認する。
- 公開ページがGitHub Pagesの場合、push直後は反映が遅れるため実配信HTMLを確認する。
- 画像遅延、ズーム、横ズレ、スクロール固定はスマホ実機で問題になりやすいので優先的に確認する。
- ドキュメントのみの変更ではアプリ内部バージョンを上げなくてよい。
- アプリ挙動やUIを変える場合は内部 `APP_VERSION` を上げる。
- 全体公開前は、更新のたびに内部 `APP_VERSION` を1つ上げ、メニュー画面のタイトル右下へ小さく表示する。
- 全体公開後は、画面上のver表示を出さない運用へ切り替える。

## 避けること

- チープな紙芝居風の結果画面。
- ただのカードUIの並び。
- 不要な説明文で画面を埋めること。
- プレイ中にスクロールできること。
- タップ領域の隙間が大きいこと。
- 画面右端や左端が見切れること。
- 同じ情報を複数箇所でくどく表示すること。
- 新記録演出をテキストだけで済ませること。
- 画像生成アセットの品質が低いままUIへ組み込むこと。

## 次回アプリ制作時の最初の依頼例

```text
AGENTS.md を読んで、エンスクドリルと同系統のUI/UXテイストで作ってください。
特に、スマホ操作性、暗い卓面風背景、明朝体、オレンジ系メインボタン、紫系高難度ボタン、結果発表演出を重視してください。
```

## 最重要判断基準

迷ったら次の順に優先する。

1. スマホで速く操作できるか
2. 誤タップやスクロール事故が起きないか
3. 画面内の情報が一目で理解できるか
4. 達成演出に気持ちよさがあるか
5. エンスクドリルの質感と連続性があるか

以上。
